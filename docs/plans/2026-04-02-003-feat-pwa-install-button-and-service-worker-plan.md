# feat: PWA install button and service worker

## Overview

Add Progressive Web App support to Zinq: a home-screen install button (house icon, top-left) that only appears in-browser, and a service worker via `vite-plugin-pwa` for app shell caching and update detection. LDK WASM stays in the main thread — the SW handles caching only.

## Problem Statement / Motivation

Zinq is a mobile-first Lightning wallet but has zero PWA infrastructure — no manifest, no service worker, no install prompt, no app icons. Users can't easily install it to their home screen, and there's no caching for faster subsequent loads or update detection for new deployments.

## Proposed Solution

### 1. Install Button

- House icon in **top-left** of Home screen (mirroring the refresh button style on the right)
- Visible only when **not** running as installed PWA (`display-mode: standalone` media query)
- On Chrome/Edge: captures `beforeinstallprompt` and triggers native install dialog
- On iOS Safari: shows manual "Add to Home Screen" instructions (since `beforeinstallprompt` doesn't exist on iOS)
- New `usePwaInstall()` hook manages the prompt state machine

### 2. Service Worker

- **`vite-plugin-pwa`** with Workbox (`registerType: 'prompt'`)
- **Precache**: app shell (HTML, JS, CSS, fonts, icons) — excludes the 12MB WASM file
- **Runtime cache**: WASM via NetworkFirst strategy (`maxEntries: 1`)
- **Update detection**: `useRegisterSW()` from `virtual:pwa-register/react` surfaces `needRefresh` state → app shows "Update available" banner
- **SPA fallback**: `navigateFallback: 'index.html'` with denylist for `/api/` routes

### 3. Web App Manifest

Generated by `vite-plugin-pwa` with:

- `name: "Zinq"`, `short_name: "Zinq"`, `display: "standalone"`
- `theme_color: "#7c3aed"` (accent purple), `background_color: "#0a0a0a"` (dark)
- Icons: 192x192, 512x512, and maskable variant — generated from existing design branding

## Technical Considerations

### Vercel Rewrite Conflict (Critical)

The current SPA rewrite `/((?!api/).*)` → `/index.html` in `vercel.json` will intercept requests for `/sw.js` and `/manifest.webmanifest`, serving HTML instead. The rewrite regex **must** exclude SW and manifest files:

```
/((?!api/|sw\.js|manifest\.webmanifest|workbox-).*)
```

**File:** `vercel.json:8`

### CSP Compatibility

The CSP in `index.html` has `script-src 'self' 'wasm-unsafe-eval'`. If `vite-plugin-pwa` injects an inline registration script, it will be blocked. Use `injectRegister: null` and manually register from `main.tsx` to avoid inline scripts.

**File:** `index.html:9`

### WASM Version Mismatch Risk

If a deployment ships new JS (precached) but the SW serves stale WASM (NetworkFirst cache), the LDK FFI bindings could mismatch and crash. Mitigation: Vite already content-hashes assets — the WASM copy script should preserve or add a hash, or the `copy:wasm` output path should be versioned. At minimum, verify that the JS glue code references the WASM by a hashed URL so each deploy requests a new cache key.

### Update Prompt Safety

For a wallet app, auto-reload is dangerous (could interrupt transactions). The update banner should:

- Only appear on the Home screen (`/`)
- Be suppressed on `/send`, `/receive`, `/scan`, and peer management routes
- Be dismissible, with re-appearance on next app open
- Also be triggerable from Settings ("Check for updates")

### iOS Limitations

- Safari does not fire `beforeinstallprompt` — the install button must detect iOS and show manual share-sheet instructions instead
- Safari evicts SW caches after ~7 days of non-use — offline caching is unreliable on iOS (acceptable for MVP)

### Offline Behavior

The cached app shell loads offline, but wallet functions require network (LDK sync, payments, peer connections). The WASM is runtime-cached (NetworkFirst), so after first load it's available offline. However, the wallet is effectively non-functional offline. No offline indicator is needed for MVP — the existing error handling in `WalletGate` covers initialization failures.

## Implementation Phases

### Phase 1: Foundation (Manifest + SW + Vercel fix)

1. Install `vite-plugin-pwa` and `workbox-window` as dev dependencies
2. Update `vercel.json` rewrite regex to exclude `/sw.js`, `/manifest.webmanifest`, `/workbox-*`
3. Configure `VitePWA()` in `vite.config.ts:56` with manifest, workbox precache (excluding WASM), runtime cache for WASM, and `navigateFallback`
4. Add `<meta name="theme-color">` and apple meta tags to `index.html`
5. Generate and add PWA icon PNGs to `public/` (192x192, 512x512, maskable, apple-touch-icon)
6. Add TypeScript types: `"vite-plugin-pwa/react"` to `tsconfig.json`
7. Use `injectRegister: null` and manually register SW from `src/main.tsx` to avoid CSP issues

**Key files:**

- `package.json` — new dev deps
- `vite.config.ts:56` — add VitePWA plugin
- `vercel.json:8` — fix rewrite regex
- `index.html:5-9` — meta tags, favicon
- `tsconfig.json` — types
- `src/main.tsx` — manual SW registration
- `public/` — icon assets

### Phase 2: Install Button

1. Add `HomeIcon` component to `src/components/icons.tsx` (after line 240, following existing `IconProps` pattern)
2. Create `src/hooks/use-pwa-install.ts` hook:
   - Listen for `beforeinstallprompt` event, store the prompt
   - Check `window.matchMedia('(display-mode: standalone)')` for installed state
   - Detect iOS via user agent for manual instructions path
   - Return `{ canInstall, isIos, promptInstall }`
3. Update `src/pages/Home.tsx:33`:
   - Change `justify-end` → `justify-between`
   - Add install button before refresh button (same `h-11 w-11 rounded-full` style)
   - Conditionally render based on `usePwaInstall()` hook
   - On iOS tap: show a simple modal/toast with share-sheet instructions

**Key files:**

- `src/components/icons.tsx:240` — HomeIcon
- `src/hooks/use-pwa-install.ts` — new hook
- `src/pages/Home.tsx:33` — install button UI

### Phase 3: Update Detection UI

1. Create an `UpdateBanner` component — simple dismissible banner ("New version available · Tap to update")
2. Use `useRegisterSW()` from `virtual:pwa-register/react` to get `needRefresh` and `updateServiceWorker`
3. Show the banner only on Home screen (check route via `useLocation`)
4. On dismiss: store in `sessionStorage`, re-show on next app open
5. On "Update" tap: call `updateServiceWorker(true)` to activate new SW and reload

**Key files:**

- `src/components/UpdateBanner.tsx` — new component
- `src/pages/Home.tsx` — render UpdateBanner conditionally

## Dependencies & Risks

| Risk                                          | Impact                         | Mitigation                                          |
| --------------------------------------------- | ------------------------------ | --------------------------------------------------- |
| Vercel rewrite intercepts SW/manifest         | SW registration fails silently | Update regex before any other work                  |
| CSP blocks inline registration script         | SW never registers             | Use `injectRegister: null` with manual registration |
| WASM/JS version mismatch after deploy         | LDK crash on cached WASM       | Verify Vite hashes WASM URL; test with cache        |
| iOS Safari evicts SW cache after 7 days       | No offline benefit on iOS      | Acceptable for MVP; app requires network anyway     |
| `beforeinstallprompt` not supported (Firefox) | Install button never shown     | Acceptable; Firefox mobile share is low             |
